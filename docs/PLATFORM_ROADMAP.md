# 平台路线图：P0 / P1 / P2

本文档定义 News Summary 从「数据管道」升级为「垂直情报简报平台」的阶段性路线，涵盖日报展示（P0）、个性化订阅（P1）、团队协作与商业化（P2）。目标客户与痛点详见 [目标客户分层](target_customers.md)，系统架构见 [系统架构设计](architecture.md)。

---

## 定位与原则

**平台定位**：高信噪比、可订阅、可推送的**垂直情报简报平台**（B2B 优先），而非泛资讯 App。

**产品原则**（优先保证）：

| 原则 | 说明 |
|------|------|
| 速度优先 | 用户打开 10 秒内看到「今天最重要 5 件事」 |
| 信噪比优先 | 宁可少，也不要公告噪音和无效重复 |
| 可信度可解释 | 每条要点能回溯原文和来源数量 |
| 个性化而非千人一面 | 让用户先定义「我关心什么」 |
| 多渠道触达 | Web 为展示层，留存靠推送 / 邮件 / IM |

---

## P0：Daily Brief 平台（4–6 周）

**目标**：在现有流水线基础上，形成统一的「日报展示 + 多渠道分发」能力，让用户「随手可看」今日重点。

### 范围与交付物

| 交付物 | 说明 | 与现有能力关系 |
|--------|------|----------------|
| Web 首页 | 今日 Top 10 事件卡，统一事件卡结构 | 依托 [frontend/index.html](../frontend/index.html) 与 [export/notebooklm.py](../backend/app/export/notebooklm.py) 的 digest 结构 |
| 事件卡统一结构 | 每个事件：发生了什么（3–5 条）、为什么重要（市场/政策影响）、多源证据链接 | 以 digest 为统一数据契约，扩展 [notebooklm.py](../backend/app/export/notebooklm.py) 字段（影响级别、地区标签等） |
| 早报邮件 | 每日固定时间发送摘要邮件 | 新增分发任务，消费同一 digest |
| 飞书 / 钉钉 / 企业微信机器人 | 推送今日摘要或按需查询 | 新增分发任务，消费同一 digest |

### 事件卡数据契约（统一接口）

所有终端（网页 / 邮件 / 机器人）消费同一 digest 结构。建议字段：

- **事件级**：`cluster_id`、标题/主题、要点列表（3–5 条）、为什么重要（影响摘要）、多源链接、分类、热度/置信度、可选：影响级别、地区标签。
- **实现**：在 [backend/app/export/notebooklm.py](../backend/app/export/notebooklm.py) 的 digest 导出上扩展上述字段，不改变现有 pipeline，仅增强输出结构。

### 质量前置（与 P0 并行或略早）

- **链接可达性**：基于 [enrich/page_meta.py](../backend/app/enrich/page_meta.py) 确保 `resolved_url` 不落到 gstatic 等无效页。
- **噪音过滤**：基于 [filter/topic_filter.py](../backend/app/filter/topic_filter.py) 压制「公司公告 / 停牌通知」类内容，提升信噪比。

### 成功指标（P0）

- 打开率（邮件 / 机器人）
- 7 日留存（是否持续查看）
- 人均阅读事件数
- 点击原文率
- 用户手动「收藏 / 转发」率（若已有点击埋点）

指标定义、计算方式与验证节奏详见 [产品成功指标与验证节奏](PRODUCT_KPIS.md)。

## P1：个性化订阅平台（8–12 周）

**目标**：用户可订阅主题、地区、关键词、公司、时间窗，获得「我的看板」与告警，只看自己关心的 20% 信息。

### 范围与交付物

| 交付物 | 说明 |
|--------|------|
| 订阅维度 | 主题（对应现有 Category）、国家/地区、关键词、公司、时间窗（如仅早报 / 每 2 小时） |
| 我的看板 | 仅展示与订阅配置匹配的事件卡，支持主题/分类导航 |
| 告警模式 | 高频更新（如每 2 小时）或重大事件即时提醒（阈值触发，如某关键词/公司出现频次） |
| 用户与配置持久化 | 账号体系（或轻量级标识）+ 订阅配置的 CRUD、与 digest 生成的联动 |

### 与现有能力的关系

- 主题订阅直接对应 [topic_filter](target_customers.md#三与系统主题分类的对应关系) 中的 `geopolitics`, `economy`, `markets`, `supply_chain`, `commodities`, `ai_tech`, `energy_infra`。
- Digest 生成逻辑不变，在输出前按用户/看板维度过滤；或对「订阅集」单独跑一轮过滤与排序。
- 需新增：用户/会话、订阅配置存储、推送调度（邮件/IM 按订阅与频率触发）。

### 成功指标（P1）

- 订阅配置率、看板打开率
- 告警打开率与 7 日留存
- 人均阅读事件数（看板内）
- 反馈闭环（有用/无用、点击原文）用于后续排序与降噪

## P2：团队协作与商业化（12 周+）

**目标**：团队共享 watchlist、点评与收藏、事件复盘；开放 API/导出；付费分层，支撑可持续运营。

### 范围与交付物

| 交付物 | 说明 |
|--------|------|
| 团队协作 | 团队空间、共享 watchlist、点评、收藏、事件复盘与备注 |
| API / 导出 | CSV / JSON / Webhook，供企业内部系统对接；与现有 [export/notebooklm.py](../backend/app/export/notebooklm.py) 导出能力对齐或扩展 |
| 付费分层 | 免费版（每日固定摘要）→ 专业版（自定义订阅 + 告警 + 历史检索）→ 团队版（协作与权限管理） |
| 历史检索 | 按时间、主题、关键词检索过往摘要与原文链接，支撑专业版/团队版 |

### 与现有能力的关系

- 导出层在现有 JSON/digest 基础上增加 Webhook、CSV 等出口；权限与用量与账号/团队绑定。
- 协作数据（watchlist、点评、收藏）需新增数据模型与 API，与 digest/事件卡 ID 关联。
- 商业化不改变核心 pipeline，仅在访问控制、限流、功能可见性上做分层。

### 成功指标（P2）

- 团队活跃度、共享 watchlist 使用率
- API 调用量、Webhook 成功率
- 免费→专业版/团队版转化率、续费率

## 落地优先级（与计划一致）

1. **先提质量，再扩平台**  
   优先修复链接可达性与噪音过滤（[page_meta](../backend/app/enrich/page_meta.py)、[topic_filter](../backend/app/filter/topic_filter.py)），再大规模优化前端与分发。
2. **统一事件卡数据契约**  
   以 digest 为统一接口，所有终端消费同一结构；在 [notebooklm.py](../backend/app/export/notebooklm.py) 扩展字段（影响级别、置信度、地区标签等）。
3. **增加分发层**  
   在现有导出流程后新增渠道：邮件、飞书、钉钉、企业微信机器人，提升「随手可看」与留存。
4. **最后补平台能力**  
   用户体系、订阅配置、历史搜索、反馈训练闭环，对应 P1/P2。

---

## 一句话策略

先打造「高信噪比、可订阅、可推送的垂直情报简报平台」，优先服务投资研究与供应链决策人群，用 P0 验证留存与价值，再用 P1/P2 扩展个性化与商业化。
