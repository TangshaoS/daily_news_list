# 产品成功指标与验证节奏

本文档定义「垂直情报简报平台」的受欢迎指标（Product KPIs）及验证节奏，用于衡量产品是否被持续使用、是否带来真实价值。与 [平台路线图](PLATFORM_ROADMAP.md) 中的成功指标一致，此处给出可执行的指标定义与校验方式。

---

## 一、核心指标定义

### 1. 打开率（Open Rate）

| 项目 | 说明 |
|------|------|
| **定义** | 在给定触达（邮件/机器人消息）中，被用户「打开」或「点击进入」的占比。 |
| **适用渠道** | 邮件早报、飞书/钉钉/企微机器人推送。 |
| **计算方式** | `打开人数（或去重打开次数）/ 送达人数 × 100%`。邮件以「邮件客户端回传打开」或「追踪像素/链接点击」近似；机器人以「消息已读/点击卡片」等平台能力近似。 |
| **目标参考** | 行业简报类邮件打开率约 20%–40%；机器人消息打开率视平台而定，可先观测基线再设目标。 |
| **数据需求** | 渠道侧需支持：送达数、打开/点击事件（或已读回执）。 |

### 2. 7 日留存（7-Day Retention）

| 项目 | 说明 |
|------|------|
| **定义** | 用户在「第 1 天」有打开/查看行为后，在第 2–7 天内至少再有 1 次打开/查看的比例。 |
| **说明** | 衡量「是否持续查看」，区分一次性使用与习惯性使用。 |
| **计算方式** | `(第 2–7 天内至少再打开 1 次的用户数 / 第 1 天有打开行为的用户数) × 100%`。可按渠道（邮件/机器人/Web）分别统计。 |
| **目标参考** | 简报/情报类产品 7 日留存 30%+ 可视为健康；冷启动期先建立基线。 |
| **数据需求** | 需能按用户/设备/邮箱标识做「首日行为」与「第 2–7 日行为」的关联（匿名或登录均可）。 |

### 3. 人均阅读事件数（Events Read per User）

| 项目 | 说明 |
|------|------|
| **定义** | 单次会话或单日内，用户实际浏览的「事件卡」数量（如展开详情、停留超过 N 秒、或滚动到底等可代理「读完」的行为）。 |
| **说明** | 衡量「是否真的看完」，而不只是打开入口。 |
| **计算方式** | `总阅读事件数（按会话或按日）/ 活跃用户数`。可先以「点击/展开事件卡数」或「详情页曝光数」为代理指标。 |
| **目标参考** | 若每日 digest 约 10–20 条事件，人均阅读 3–5 条以上可认为参与度较好。 |
| **数据需求** | 前端或邮件需埋点：事件卡曝光/展开/点击原文；无用户体系时可按 session 或 device 近似。 |

### 4. 点击原文率（Click-Through to Original, CTR）

| 项目 | 说明 |
|------|------|
| **定义** | 用户在阅读摘要/事件卡后，点击「原文链接」进入源站的比例。 |
| **说明** | 反映摘要是否可信、是否有价值，用户是否愿意进一步深读。 |
| **计算方式** | `点击原文次数 / 事件卡曝光数（或阅读事件数）× 100%`；或 `至少点击过 1 次原文的用户数 / 有阅读行为的用户数 × 100%`。 |
| **目标参考** | 情报/简报类产品，单事件卡点击原文率 5%–15% 较常见；可先观测基线再设目标。 |
| **数据需求** | 所有原文链接需带追踪参数或经统一跳转，以统计点击；前端与邮件/机器人链接需统一处理。 |

### 5. 收藏/转发率（Save or Forward Rate）

| 项目 | 说明 |
|------|------|
| **定义** | 用户对单条事件或整份简报执行「收藏」或「转发」的比例。 |
| **说明** | 衡量信息价值密度与用户主动传播意愿。 |
| **计算方式** | `收藏或转发次数 / 事件卡曝光数（或用户会话数）× 100%`；或 `有收藏/转发行为的用户数 / 活跃用户数 × 100%`。 |
| **目标参考** | 高价值情报场景下，该指标通常不高但极具参考价值；冷启动期可先做埋点与采集，再定目标。 |
| **数据需求** | 需提供「收藏」「转发」入口（Web/邮件/IM），并上报行为事件；无该功能前可暂不统计。 |

| 指标 | P0 Daily Brief | P1 订阅/看板 | P2 团队/商业化 |
|------|----------------|-------------|----------------|

## 三、验证节奏（Validation Cadence）

### 3.1 数据采集节奏

| 类型 | 频率 | 说明 |
|------|------|------|
| **埋点与日志** | 实时/准实时 | 所有与上述指标相关的事件（打开、点击、展开、收藏、转发）应持续上报，便于按日/周/月聚合。 |
| **渠道回传** | 按渠道能力 | 邮件打开/点击、机器人已读或点击，依各渠道 API 或 webhook 尽量同步回传。 |
| **聚合表/快照** | 每日 | 建议每日 EOD 生成「按日」的指标快照（打开率、人均阅读、点击原文率等），便于趋势对比。 |

### 3.2 指标审视节奏

| 审视类型 | 频率 | 动作 |
|----------|------|------|
| **日报/周报看板** | 每日或每周 | 查看打开率、人均阅读事件数、点击原文率的日/周趋势；异常日标注原因（如推送失败、内容空窗等）。 |
| **留存专项** | 每周 | 计算当周「新打开用户」的 7 日留存曲线，观察是否改善或恶化。 |
| **KPI 复盘会** | 每两周或每月 | 回顾 5 项核心指标与目标/基线的差距；决定是否调整产品（如推送时间、摘要长度、噪音过滤）或运营动作。 |
| **基线建立** | 上线后 2–4 周 | 在未大改产品的前提下，先跑出各指标的基线区间，再设定「提升 X%」类目标。 |

### 3.3 验证与产品迭代的闭环

1. **假设**：例如「缩短邮件标题能提高打开率」「增加事件卡摘要长度会提高点击原文率」。  
2. **改版**：在单渠道或部分用户做小范围改动。  
3. **观测**：按上述节奏看对应指标 1–2 周内的变化。  
4. **决策**：达标则全量或保留；未达标则回滚或再迭代。

## 四、实施优先级（与现有能力对齐）

| 阶段 | 可立即做的 | 依赖能力补齐后做的 |
|------|------------|---------------------|
| **当前（无推送、无账号）** | 若已有 Web 访问：可埋点「事件卡曝光/展开」「点击原文」，先算人均阅读事件数、点击原文率；定义好指标口径与表结构。 | 打开率、7 日留存、收藏转发依赖「渠道推送」与「用户/设备标识」。 |
| **P0（有邮件/机器人后）** | 打开率、7 日留存（按邮箱或设备）、人均阅读、点击原文率；链接统一跳转或带参以便统计。 | 收藏/转发需产品入口与埋点。 |
| **P1/P2** | 按订阅/团队维度拆分上述指标；增加「订阅配置率」「告警打开率」等。 | 与用户体系、权限、套餐一致。 |

---

## 五、一句话小结

**指标**：打开率、7 日留存、人均阅读事件数、点击原文率、收藏/转发率，五者共同衡量「受欢迎」与「有价值」。  
**节奏**：埋点持续采集 → 每日/周看板审视 → 每两周/月 KPI 复盘 → 用数据驱动推送策略、内容质量与产品形态迭代。
